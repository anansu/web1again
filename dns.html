Domain Name System

Jon Postel과 Paul Mockapetris에 의해 DNS가 탄생함. DNS의 핵심은 DNS서버임.

IP주소와 host 개념

도메인 대신 쓸 수 있는 간단한 방법 먼저 확인해보기. 인터넷에 연결된 모든 컴퓨터들은 각자 IP주소를 갖고 있어야 함. 이 각각에 인터넷에 연결된 컴퓨터(장치)들을 host라고 부르기도 함. (host=네트워크에 연결된 컴퓨터, 장치들) 93.184.216.34 같은 컴퓨터에 클라이언트로서 접속하려면, IP주소를 기억하기 빡세다. 그래서 example.com쳐도 들어가고 싶다 이말이야. 다행스럽게도 모든 운영체제에는 hosts라는 파일이 있음. 이 파일에다가 example.com의 주소는 93.184~라고 써두면 example.com을 입력해도 잘 들어가진다. 이게 hosts라는 파일의 역할. DNS를 통하지 않고도 잘 들어갈 수 있다는 것. hosts 파일을 통해 내가 지정한 주소-이름명을 맘대로 조작할 수 있음. 예를 들어 hosts파일에다가 93.184.216.34 주소를 naver.com으로 해둔다? 그러면 naver.com을 쳐도 93.184.216.34로 들어가게 된다. 이 hosts파일을 악의를 가지고 변조당하게 되면, 위험하다 위험해.(의도치않은 사이트로 들어갈 수 있다...) 이런걸 피싱이라고 함. 굉장히 위협적인 공격임. hosts 파일을 잘 관리해야함. https로 끝난다? Secure - 암호화되어있는 곳. http로 접속하면 변조되어도 알 수 없음. 그러나 https로 끝나면 안심하고 볼 수 있다. 훨씬 더 안전함.

DNS의 태동

before DNS를 알아야 DNS를 이해하기 편하다. DNS이전의 인터넷이 어떻게 동작했는지 확인해보자. 이전: hosts라는 파일을 개인적으로 운영했음. example.com을 특정주소에 연결해놔도 어차피 전부 본인한테만 해당된다고. 그럼 관점을 바꿔서, IP를 갖고 있는 사람은 example.com으로 고정해두고 싶을 것. 이걸 하기 위해 중간에 신뢰할 수 있는 매개자가 필요했음. 이걸 처음 한 게 Stanford Research Institute, 요기가 중간 관리자로 등록해서 쓰곤 했음 그러면 SRI에 있는 전화번호부(hosts 파일)을 다운받아서 사용했음. 그러나 이건 인터넷이 커질수록 문제가 커졌음. 이런 문제를 극복하기 위해 1983년, 인터넷의 신이라 불린 Jon Postel과 Paul Mockapetris가 만든게 DNS!

DNS의 원리

DNS 서버에, 자신의 ip는 ~인데 이 ip는 ~~~라는 이름을 갖게 하고 싶다는 요청을 DNS서버에다가 하는거임. 그러면 DNS서버는 그 요청을 받고 도메인네임과 ~의 연결을 기억해둠. 자 이제 우리 컴터가 example.com이라는 도메인네임으로 접속하려고 하면 집에 있는 랜선/와이파이 등에 마법같은 메커니즘에 의해서 우리 컴퓨터에는 도메인 네임 시스템 서버의 ip주소가 자동으로 세팅됨. (by DHCP) 그럼 인터넷 연결한 순간 DNS 서버의 ip를 알기 때문에, 우리 컴퓨터는 약속된대로 hosts파일을 찾아보고, 그 다음에 DNS 서버에 접속해서 example.com이라는 네임의 ip가 뭐냐고 물어봄. 그러면 dns서버는 기억하는 주소를 응답해줌. 그러면 알게된 ip주소로 접속해서 example.com과 통신하게 되는 것. 우리가 클라이언트로 dns를 활용할 때 필요한 적정지식이 여기까지. dns 서버를 이용하는 모든 컴들은 변경된 사항을 즉시 반영가능하게 됨.

public DNS의 사용

dns서버도 ip를 갖고 있어야만 여기에 접속할 수 있겠지. 그런데 example.com이라고 쳤을 때 dns 서버에 접속해서 example.com의 ip를 물어봐서 가져올 수 있다고 이야기했음. 그러려면 이 컴퓨터(클라이언트가)가 Dns 서버의 ip나 도메인을 알고 있어야 함. 그걸 어떻게 알 수 있냐? 이 컴퓨터를 인터넷에 연결하는 순간 우리가 가입한 통신사(ISP)들이 우리의 컴퓨터에다가 "도메인을 물어봤을 때 사용할" DNS이전의 ip를 자동으로 세팅해줌. 그래서 연결하는 순간 자동으로 이 컴퓨터(클라이언트)가 dns 서버를 쓰게 된다든 것. 근데 우리가 통신사가 지정해준 dns서버가 아니라 다른 dns서버를 써보고 싶을 수 있음. 통신사가 우리가 접근하는 사이트의 정보를 저장하거나 할 수 있기 때문에 기분나쁠 수도 있다 이말이야. 심각한 privacy 문제가 있을 수 있음. 그런 경우 더 빠른, 더 우리 입장에서 안전한 DNS서버를 사용하고 싶을 수 있다. 우린 이미 이걸 할 수 있음. 엔진에서 검색해보면(public dns server) 목록이 많음. 대표적으로 구글에서 public dns를 만들어서 무료로 사용하도록 open한적 있음. dns설정에서 dns를 8.8.8.8로 하면, 여기에 해당하는 서버컴터의 dns서버에 도메인을 물어보게 된다는 것. 그렇게 우리가 쓰는 dns서버를 다른걸로 바꿀 수 있다는 것. 1.1.1.1 이라는 public dns도 나왔다. 기관-기업이 힘 합쳐서 만든 public dns임. 프라이버시도 챙기고... dns 서버 설정하는것만 네트워크에서 어댑터 설정에서 1.1.1.1. 보조 dns 1.0.0.1로 바꾸니 이거저거 다 된다고 함. 일단 다 되었다.

도메인 이름의 구조

DNS의 메커니즘, 내부 구조를 살펴보겠음. 클라이언트로서만 인터넷을 사용한 경우는 안 봐도 됨. 그러나 서버를 운용한다면? 지금부터 설명하는 것이 중요한 부분. DNS의 원리 - DNS서버가 한대가 있는데, 한대가 하는 역할을 2가지로 분리할 수 있음. (1) 서버로 사용할 컴퓨터가 자신의 이름-ip를 제출해서 등록하는 것. (2) 클라로 사용할 컴퓨터가 이름을 물어보면 ip를 알려주는 것. 즉 기억하는 기능 + 물어보면 알려주는 기능을 하는 1대의 기계로 그려져 있음. dns서버는 전 세계에 수천~수만대의 dns서버가 흩어져있음. 전세계인들이 ip주소 입력안하고도 각자 연결가능하게 돕고 있다 이말이야. 도메인의 주소 뒤에는 .이 생략되어 있음. naver.com. 이런 식으로 이 도메인서 각각의 부분마다 이름이 있음. 마지막 점 = 루트 도메인서그 밑에는 .com이 있는데, 이걸 Top level 도메인이라 함. 뭐 .net .kr 등이 전부 top level 도메인임. 그 담에 나오는 example/naver같은 것들이 second-level 도메인이라고 함. 나머지를 sub-domain이라고 함. 즉 sub - second-level - top-level - root 이 각각의 부분에 대해 각각을 담당하는 독자적인 서버가 존재함. 이 루트 도메인 담당하는 dns서버는 그 하위라 할 수 있는 탑-레벨 도메인을 담당하는 서버들의 목록을 알고 있어야 함. 그리고 탑 -레벨을 담당하는 dns서버들은 세컨레벨 서버들을 알고있고, 세컨드는 서드의 목록과 ip주소를 알고 있어야 함. 자기 직속 하위 부서만 알고 있는 것임. 예) blog.example.com. 이라고 하면.. sub second tope root 이게 우리 컴퓨터가 blog.example.com이라는 주소를 조회해서 ip주소를 알고 싶다? sub도메인을 전담하고 있는 dns서버가 blog.example.com. 의 ip주소를 알고 있음. 한방에 저 sub도메인 전담하고있는 dns서버를 한방에 알 수 없기 때문에 1. Root 네임 서버의 ip주소가 뭔지는 알고 있음. root name 서버는 top level이 com이니까 top lv dns 서버에 물어보라고 함.(com을 담당) 그러면 example을 담당하는 second lv 서버에 물어보게 되고 그에 따라 sub로 가게 된다. 이 과정을 통해 컴퓨터는 원하는 ip주소를 알 수 있게 됨.
